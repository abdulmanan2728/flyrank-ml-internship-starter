Abstract

Content teams can't manually review every page for decline every week — so which pages should an analyst check first? This work builds a scoring model on FlyRank's pseudonymized content-performance dataset (30,000 pages, 32 clients, trailing-90-day search metrics) to flag pages showing a downward 30-day impression trend. A Random Forest trained on 28 engagement and freshness features is validated against a client-grouped holdout — never a naive random split, which this paper shows inflates results by roughly 10 points of precision. Under the honest split, the model reaches 0.805 precision and 0.900 recall, against a hand-written baseline rule's 0.524 precision and 0.100 recall on the same test rows. The result is packaged as a ranked, reason-coded weekly review queue — a decision-support tool for prioritizing human attention, not a claim about causation or about Google's ranking algorithm.

1 · Introduction

The problem isn't finding declining pages. It's finding them in time.

Search-visible content decays unevenly across a client's page set, and manually checking every page for decline doesn't scale once a portfolio reaches thousands of pages. This work supports a simple, recurring decision: which pages should an SEO analyst review this week, and why?

The output isn't a verdict — it's a ranked, reason-coded queue that an analyst works through, top-down, until their review capacity for the week runs out. A false positive here costs a few wasted minutes reviewing a page that turned out fine. A false negative costs a real page another week of unnoticed decline. Neither failure is catastrophic, which is exactly why this is framed as decision-support, not an automated action.

A single hand-written rule — "flag any page with below-median CTR," say — turns out to be far too blunt for this. Decline is driven by a tangled combination of momentum, click behavior, and freshness that shifts across content types and clients; the sections below show, with real numbers, exactly how much a learned model captures that a fixed rule misses.

This isn't a hypothetical problem. FlyRank's own March 2026 portfolio research independently confirmed the pattern this capstone targets: health score climbs from 7 at day 0–7 to a peak of 33.1 at 61–90 days, then falls to 14 by 271–365 days — a real decay cliff across 341,701 real content pieces. The same research found that refreshing mature, previously-visible content produced a 3.2x health lift and 57x more impressions. This capstone asks a narrower, complementary question: rather than confirming that refresh works in aggregate, can a model tell an analyst which specific pages, out of a large portfolio, are worth that refresh effort this week?

2 · Data

Data

Release: FlyRank's pseudonymized starter dataset — content_refresh_anonymized.csv — 30,000 content items across 32 clients, each row a trailing-90-day performance snapshot.

Unit of analysis: one row = one pseudonymized content item. Grain was verified directly — zero duplicate content_id values across all 30,000 rows.

Label: is_declining, built from the dataset's own trend_direction field (trend_direction == 'down') — an observed 30-day-vs-prior-30-day impression comparison, not a hand-labeled judgment. Base rate: 0.542 (54% of pages trending down), close to balanced.

Deliberately excluded as features: trend_pct and trend_direction themselves are the source of the label — using either as a model input would let the model learn the label's own definition rather than a genuine predictive pattern. Every candidate feature was checked against trend_pct for correlation before inclusion; the strongest correlation found was 0.097, well below any concerning threshold.

Public-safety: no client names, domains, URLs, or raw queries appear anywhere in this paper or the underlying notebooks — all identifiers are FlyRank's pseudonymized hash keys.

3 · Methodology

Methodology

Features

28 features spanning traffic volume (impressions_90d, clicks_90d, 30-day windows), engagement (scroll_rate, engagement_rate), freshness (days_since_last_update, content_age_days), and ranking position (avg_position, ctr) — every one available at decision time, none derived from the label.

Baseline

A transparent, hand-written rule, built the same way FlyRank's own internal flags are built: flag a page if it's both stale (31–180 days since last update — the 181+ tier was excluded after a signal check showed decline rate actually reverses downward past 180 days) and has below-median click-through rate relative to its position-tier peers. Both signals were verified independently before use — staleness gave a mixed verdict (decline rate rises through 91–180 days, then reverses for 181+), while the CTR-vs-position gap gave a clean confirmed verdict (0.593 vs. 0.503 decline rate across large samples).

Model

Random Forest — chosen because decline isn't driven by one clean signal but by tangled, nonlinear interactions between the features above, which tree ensembles capture without hand-specifying interactions, while still producing interpretable feature importances.

Validation design

Split grouped by client_id, not random — no client's pages appear in both train and test. This matters concretely: a naive random split on this exact model produced 0.901 precision, a number inflated by the model partly memorizing client-specific patterns. The grouped split gives the honest number: 0.805 precision. That ~10-point gap is reported here directly rather than the flattering number alone.

Leakage audit

Every feature's correlation with the label's source column (trend_pct) was checked twice, independently, across two separate audits. Maximum correlation found: 0.097. No label-derived column appears in the feature set at any point.

0.901 is the number a careless evaluation would report. 0.805 is the number that actually reflects performance on a client the model hasn't seen before — the number this paper stands behind.

What the model leans on

  • impressions_prev_30d — 0.211 importance
  • impressions_last_30d — 0.158
  • impressions_90d — 0.080
  • avg_position — 0.062
  • days_with_impressions — 0.046

Recent impression momentum dominates — sensible, since the label itself is fundamentally about impressions changing.

Where it's wrong

False positives (800 rows, 11.2% of test set) outnumber false negatives (365 rows, 5.1%) more than 2-to-1 — the model errs toward flagging borderline cases, consistent with a deliberate precision-over-recall design choice. False positives skew toward lower-traffic pages (avg. 2,160 impressions vs. 3,728 overall) — a page with few impressions produces noisier percentage swings that are easier to misread as decline.

5 · Limitations & honest framing

What this can and cannot claim

Can claim: under a client-grouped holdout, the model achieves 0.805 precision and 0.900 recall at flagging pages with a downward 30-day-vs-prior-30-day impression trend — a directional, decision-support signal for prioritizing manual review.
Cannot claim: that this predicts future outcomes with certainty, that any one feature causally drives decline, or anything about Google's ranking algorithm. High feature importance describes what the model leans on — it is not evidence of an external cause.
  • Trained on a single historical snapshot — doesn't account for seasonality or algorithm changes since the data was captured.
  • review + review_urgent flags cover 50.5% of the full inventory — far more than any team can review weekly. risk_score is a ranking signal to sort within capacity, not a fixed workload.
  • Validated on held-out clients from the same snapshot, not on genuinely new data collected after training — real-world drift is untested.
  • The starter dataset (30K rows) is a slice of FlyRank's much larger warehouse (~79M rows); results here should be read as a validated prototype, not a claim about the full population.

6 · Ranked recommendations

The action playbook

review_urgent · 5,525 pages review · 9,627 pages monitor · 2,442 pages healthy · 12,406 pages
  1. Start with review_urgent: high risk score AND stale (>90 days since update).
    Pairs the model's strongest signal with the freshness pattern that FlyRank's own research has independently confirmed — decline accelerates sharply once content passes roughly nine months untouched.
  2. Rank review by risk_score and take only the top slice your team has capacity for.
    50.5% of inventory is flagged in total — far more than a weekly workload. Treat the score as a sort order, not a checklist to clear in full.
  3. Before acting on any flag, check impression volume.
    False positives skew toward low-traffic pages — a flag on a page with under ~20 impressions deserves a sanity check before spending real effort.
  4. Never auto-publish, auto-unpublish, or auto-deindex based on the score alone.
    The score is a review prompt built from proxy features, not a validated quality verdict — every action needs a human read on the actual page.
  5. Re-validate quarterly, or sooner if the flagged share drifts sharply from 50.5%.
    The model was trained on one historical snapshot; portfolio composition and search behavior both shift over time.
